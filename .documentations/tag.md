## Большая картина: зачем вообще эта система

Тебе нужно показывать теги на карточках программ — "Windows", "Бесплатно", "Графика" и т.д. Но теги нужно группировать, переводить на разные языки, и чтобы у каждого тега была своя страница в поиске Google.

---

## Уровень 1 — Категории и Теги (полки и этикетки)

Представь книжный магазин. Есть **полки** — "Операционные системы", "Лицензия", "Тип приложения". На каждой полке лежат **книги с этикетками** — "Windows", "macOS", "Бесплатно", "Платно".

В коде это:

* **CategoryTag** = полка. У неё есть цвет (синяя полка) и иконка.
* **Tag** = этикетка на книге. Привязана ровно к одной полке.

Тег **обязан** принадлежать категории — нельзя создать тег "в воздухе".

---

## Уровень 2 — Наследование цвета и иконки

Допустим, категория "Лицензия" — зелёная с иконкой 💰.

Ты создаёшь тег "Бесплатно" и не задаёшь ему цвет — он автоматически становится зелёным с 💰, как у родителя.

Но тег "Enterprise" ты хочешь сделать золотым — просто задаёшь ему свой цвет, и наследование отключается именно для него.

Это происходит  **не в базе данных** , а в момент отдачи данных клиенту — в маппере. В базе лежит буквально строка `"inherit"`, и маппер при чтении подставляет цвет категории.

---

## Уровень 3 — Переводы и красная точка

У каждого тега и категории есть переводы — отдельная таблица с парами "язык + текст". Например:

* RU → "Бесплатно"
* EN → "Free"
* DE → "Kostenlos"

Система знает, сколько языков активно (допустим, 3). Если у тега только 2 перевода — в админке рядом с ним появляется  **красная точка** . Это сигнал: "эй, переведи на немецкий".

Это вычисляется каждый раз при запросе, ничего не хранится — просто сравниваются два числа.

---

## Уровень 4 — SEO страницы тегов

Тег "Windows" — это не просто этикетка. У него есть своя страница `/tag/windows` с заголовком, мета-описанием и H1. Это хранится в таблице локализаций прямо рядом с переводом названия.

Пользователь гуглит "лучшие программы для Windows" — попадает на страницу тега, а оттуда уже видит все программы с этим тегом.

---

## Уровень 5 — Связь с программами

Программа может иметь несколько тегов. Это хранится в отдельной bridge-таблице:

```
Photoshop → [Windows, macOS, Платно, Графика]
```

У каждой связи есть два поля: **SortOrder** (в каком порядке показывать теги на карточке) и **IsMain** (один тег — главный, он показывается даже на мини-карточке в списке).

---

## Уровень 6 — Защита от случайных поломок

Если ты пытаешься удалить категорию "Лицензия", а в ней 50 тегов — система не даст это сделать. Вернёт ошибку с человекочитаемым текстом, и кнопка удаления в UI будет серой с подсказкой "сначала перенеси или удали теги".

Это защита от ситуации когда удалил категорию → 50 тегов повисли без привязки → карточки программ сломались.

---

## Как это выглядит для пользователя

1. Администратор создаёт категорию "ОС" с синим цветом
2. Создаёт теги "Windows", "macOS", "Linux" — они все автоматически синие
3. Переводит каждый тег на все языки
4. Привязывает теги к программам
5. Пользователь на сайте видит синие бейджи на карточках программ и может фильтровать

Вот и всё — ничего магического, просто аккуратно выстроенная иерархия с несколькими защитными механизмами.

3. Бизнес-логика (Service Layer)

3.1. Стратегия наследования визуальных свойств

Реализуется в TagMapper (AutoMapper) при формировании DTO. В базе данных хранятся «сырые» значения; вычисленные DisplayColor и DisplayIcon существуют только в DTO.

// Цвет

dest.DisplayColor = (src.Color == "inherit" || string.IsNullOrEmpty(src.Color))

    ? src.Category.Color

    : src.Color;

// Иконка

dest.DisplayIcon = string.IsNullOrEmpty(src.IconName)

    ? src.Category.IconName

    : src.IconName;

3.2. Delete Guard — защита удаления категории

FK между Tag и CategoryTag настроен как DeleteBehavior.Restrict. Прямое удаление категории с активными тегами запрещено на уровне БД. Дополнительно Guard реализуется в сервисе.

// CategoryTagService.DeleteAsync(int id)

var hasTags = await _db.Tags

    .AnyAsync(t => t.CategoryTagId == id && !t.IsDeleted);

if (hasTags)

    throw new BusinessException(

    "CATEGORY_NOT_EMPTY",

    "Нельзя удалить категорию, содержащую теги.");

| 👁️ UX | API возвращает HTTP 409 Conflict с кодом CATEGORY_NOT_EMPTY.Frontend отображает Toast-уведомление: «Сначала переназначьте или удалите теги этой категории».Кнопка удаления в UI категории с тегами — отображается disabled с Tooltip-объяснением. |
| ------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

3.3. Сквозной поиск (Cross-Language Search)

Поиск выполняется через IQueryable с использованием Any() по коллекции локализаций. Требует GIN-индекса (см. раздел 2).

query.Where(x =>

    x.Slug.Contains(searchTerm) ||

    x.Localizations.Any(l => l.Name.Contains(searchTerm)));

3.4. Статус RequiresTranslation

Вычисляемое поле в DTO. В UI отображается как красный индикатор (Badge / Dot).

dto.RequiresTranslation =

    entity.Localizations.Count < _languageService.GetActiveLanguagesCount();

3.5. Управление SortOrder

При создании новой записи: SortOrder = (db.Set.Max(x => x.SortOrder) ?? 0) + 1.

| ⚠️ РИСК | Паттерн Max+1 может порождать дубликаты при параллельных запросах (race condition). Для MVP это приемлемо.При росте нагрузки заменить на PostgreSQL SEQUENCE или добавить UNIQUE constraint на (SortOrder, IsDeleted) с обработкой конфликта на уровне сервиса. |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

Пакетная сортировка (Drag-and-Drop): метод UpdateSortOrderAsync(List `<SortItemDto>` items) обновляет массив ID в одной транзакции.

5. Fluent API — ключевые настройки

Всё ниже реализуется в AggregatorModelConfiguration.cs (или отдельных файлах конфигурации).

// CategoryTag → Unique Slug

builder.HasIndex(x => x.Slug).IsUnique();

// Tag → Unique Slug

builder.HasIndex(x => x.Slug).IsUnique();

// Tag → Composite index

builder.HasIndex(x => new { x.CategoryTagId, x.IsActive, x.IsDeleted });

// Tag → FK с Restrict

builder.HasOne(t => t.Category)

    .WithMany(c => c.Tags)

    .HasForeignKey(t => t.CategoryTagId)

    .OnDelete(DeleteBehavior.Restrict);

// CategoryLocalization → FK без каскада

builder.HasOne(l => l.CategoryTag)

    .WithMany(c => c.Localizations)

    .HasForeignKey(l => l.CategoryTagId)

    .OnDelete(DeleteBehavior.NoAction);

// ProgramTag → составной PK

builder.HasKey(x => new { x.ProgramId, x.TagId });

// GIN-индекс (raw SQL в миграции)

migrationBuilder.Sql(

    "CREATE INDEX ix_localizations_name_gin"

    + " ON tag_localizations USING GIN (\"Name\" gin_trgm_ops);");

7. Инициализация данных (Seeding)

Для наполнения системы базовыми тегами используется JSON-файл.

### 7.1. Формат JSON (tags-seed.json)

Обеспечивает иерархическую загрузку: сначала Категория, затем её Теги.

```json
{
  "Categories": [
    {
      "Slug": "license-types",
      "IconName": "lucide-shield-check",
      "Color": "var(--av-tag-purple)",
      "SortOrder": 1,
      "Translations": {
        "ru-RU": { "Name": "Типы лицензий", "Description": "..." },
        "en-US": { "Name": "License Types", "Description": "..." }
      },
      "Tags": [
        {
          "Slug": "freeware",
          "Type": "License",
          "Color": null, 
          "IconName": null,
          "SortOrder": 1,
          "IsFeature": true,
          "Translations": {
            "ru-RU": { "Name": "Бесплатно", "MetaTitle": "...", "H1Title": "..." },
            "en-US": { "Name": "Freeware", "MetaTitle": "...", "H1Title": "..." }
          }
        }
      ]
    }
  ]
}
```

### 7.2. Логика работы Seed-сервиса

1. **Ключ идентификации**: Slug.
2. **Алгоритм (Upsert)**: Найти по Slug. Если найдено — обновить (Update), если нет — создать (Insert).
3. **Локализация**: Обновление/создание локализаций для всех указанных BCP-47 кодов.
4. **Валидация**: Отсутствие en-US не блокирует загрузку, но логируется как Warning в консоль.
