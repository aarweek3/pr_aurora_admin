## 3. Код ответа сервера (Backend Service Logic)

**Файл**: `Project_Server_Auth/Pages/AGGREGATOR/CategoryOfAggregator/Services/CategoryOfAggregatorService.cs`

```csharp
public async Task<List<CategoryOfAggregatorItemDto>> GetTreeAsync(int? languageId = null)
{
    // 1. Выборка ВСЕХ записей из БД одним списком
    var allItems = await _unitOfWork.CategoriesOfAggregator.GetTreeAsync(languageId);

    // 2. Маппинг в DTO
    var dtos = allItems.Select(item => {
        var dto = _mapper.Map<CategoryOfAggregatorItemDto>(item);
        var loc = item.Localizations.FirstOrDefault(l => l.LanguageOfAggregatorId == languageId)
                  ?? item.Localizations.FirstOrDefault();
        dto.LocalizedName = loc?.Name;
        return dto;
    }).ToList();

    // 3. Рекурсивное построение дерева в памяти
    var rootNodes = dtos.Where(d => d.ParentId == null)
                        .OrderBy(x => x.LocalizedName ?? x.CanonicalName)
                        .ToList();
    foreach (var root in rootNodes)
    {
        BuildTree(root, dtos, 0);
    }

    return rootNodes;
}

private void BuildTree(CategoryOfAggregatorItemDto parent, List<CategoryOfAggregatorItemDto> all, int level)
{
    parent.Level = level;
    parent.Children = all.Where(x => x.ParentId == parent.Id)
                        .OrderBy(x => x.LocalizedName ?? x.CanonicalName)
                        .ToList();
    foreach (var child in parent.Children)
    {
        BuildTree(child, all, level + 1);
    }
}
```

## Краткое пояснение (Как это работает)

1. **Хранение**: В базе данных используется классический `ParentId` (Adjacency List). Также есть поля `HierarchyPath` и `Level` для материализованного пути, но текущая логика чтения их практически не использует.
2. **Загрузка**: При вызове `getTree` сервер выкачивает **всю таблицу** категорий целиком (`CategoryOfAggregatorRepository.GetTreeAsync` делает просто `Entities.ToListAsync()`).
3. **Сборка**: После получения плоского списка, сервис запускает рекурсивный метод `BuildTree`, который для каждого узла ищет его детей в общем списке `dtos.Where(x => x.ParentId == parent.Id)`.
4. **Проблема**: При большом количестве категорий (например, >5000) загрузка всей таблицы и последующая рекурсивная обработка создают высокую нагрузку на RAM и увеличивают время ответа (TTFB).

# План оптимизации производительности CategoryOfAggregator

Этот план направлен на устранение критических проблем (N+1, O(N²), избыточный трафик и DOM-нагрузка) при работе с деревом категорий (5000+ записей).

---

## Фаза 0 — Подготовка данных (SQL)

Заполнение полей `HierarchyPath` и `Level`, которые уже есть в модели, но не используются.

**0.1 Скрипт миграции**

- **Действие**: Рекурсивно вычислить `HierarchyPath` (напр. `/1/5/12`) и `Level` для всех записей.
- **Безопасность**:
  - Обязательный бэкап таблицы перед запуском.
  - Защита от бесконечных циклов: `WHERE @depth < 20`.
- **Результат**: Поля готовы к использованию в SQL-индексах и логике перемещения веток.

---

## Фаза 1 — Быстрые победы (Backend)

**1.1 Внедрение CategoryTreeBuilder (Helper)**

- **Логика**: Вынести сборку дерева из `CategoryOfAggregatorService` в статический хелпер.
- **Алгоритм**: Использовать `ToLookup(x => x.ParentId)` для мгновенного поиска детей. Сложность сборки: **O(N)** вместо текущей **O(N²)**.
- **Преимущество**: Сервис становится «тонким», а логику дерева легко тестировать юнит-тестами.

**1.2 Интеллектуальный IMemoryCache**

- **Ключ**: `tree_{languageId}`.
- **Правило**: Кэшируем только основное дерево (`showDeleted = false`).
- **Корзина**: Запросы с `showDeleted = true` **не кэшируются**. Это редкий административный сценарий, требующий 100% актуальности.
- **Инвалидация**: Полный сброс или пересчет кэша при любых изменениях (Create/Update/Delete).

**1.3 Исправление N+1 на уровне SQL**

- **Действие**: В `CategoryOfAggregatorRepository.GetTreeAsync` добавить `.Include(x => x.Children)` или использовать проекцию `.Select()` для подсчета `ChildrenCount`.

---

## Фаза 2 — Архитектурные изменения (Backend)

**2.1 Lazy Loading (Поуровневая загрузка)**

- **API**: `GET /tree?parentId={id}`.
- **Логика**: Возвращать только прямых потомков указанного узла. Если `parentId` не передан — возвращать только корни.
- **Эффект**: Трафик снижается с **200кб+** до **2-5кб** на запрос.

**2.2 Обновление HierarchyPath (Вариант А — Batch Update)**

- **Сценарий**: При смене `ParentId` у категории.
- **Реализация**: Одним SQL-запросом через `REPLACE(hierarchy_path, @oldPath, @newPath)` обновить пути у всей ветки (категория + все её потомки).
- **Обоснование**: Категории меняются редко, поэтому сложная логика при записи оправдана скоростью чтения.

**2.3 Серверный поиск с AncestorIds**

- **Логика**: Поиск по дереву возвращает найденные элементы и массив всех их родителей (`AncestorIds`).
- **Зачем**: Позволяет фронтенду в режиме Lazy Loading автоматически раскрыть дерево до найденного элемента.

---

## Фаза 3 — Фронтенд (Angular/UI)

**3.1 Виртуализация (Virtual Scroll)**

- **Компонент**: `nz-tree` или `nz-table` с параметром `[nzVirtualScroll]="true"`.
- **Эффект**: Браузер рендерит только ~20 видимых элементов. DOM остается легким независимо от размера дерева.

**3.2 Клиентское кэширование**

- **Реализация**: В `CategoryOfAggregatorApiService` хранить уже загруженные ветки в `Map<number | null, CategoryDto[]>`. Повторное раскрытие узла происходит мгновенно без сетевого запроса.

---

## Фаза 4 — Индексация

- `CREATE INDEX idx_category_hierarchy ON categories_of_aggregator(hierarchy_path);`
