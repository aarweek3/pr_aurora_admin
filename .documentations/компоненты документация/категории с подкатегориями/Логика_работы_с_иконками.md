# Логика работы с иконками (Simplified Categories)

В данном документе описана техническая реализация системы управления иконками в модуле упрощенных категорий, включая процессы загрузки, добавления и редактирования.

Иконки и начальные данные (seed) не хранятся в базе данных. Они управляются как статические ресурсы через JSON-файлы на бэкенде:

- **Иконки (SVG)**: `Project_Server_Auth/Pages/AGGREGATOR/CategorySimplified/Jsons/category-icons.json`
- **Данные категорий (Seed)**: `Project_Server_Auth/Pages/AGGREGATOR/CategorySimplified/Jsons/categories-seed.json`

### Структура файла иконок:
```json
[
  {
    "name": "icon-name",
    "svgCodIcon": "<svg>...</svg>"
  }
]
```

## 2. Загрузка иконок (Loading)

### Бэкенд
Метод `GetIconsAsync` в `CategorySimplifiedService` считывает содержимое JSON-файла, десериализует его в список DTO (`CategoryIconDto`) и возвращает фронтенду.

### Фронтенд
1.  При инициализации страницы или открытии модального окна `CategorySimplifiedStateService` вызывает `loadIcons()`.
2.  Данные сохраняются в сигнале `customIcons`.
3.  Для быстрого поиска по имени автоматически вычисляется `iconsMap` (Computed Signal), который используется компонентами рендеринга.

```typescript
// CategorySimplifiedStateService
loadIcons(): void {
  this.api.getIcons().subscribe({
    next: (icons) => this.updateState({ customIcons: icons }),
    error: (err) => console.error('Failed to load icons', err)
  });
}
```

## 3. Добавление новых иконок (Adding)

Добавление новой иконки происходит через административный интерфейс (обычно в модальном окне управления иконками):

1.  **Ввод данных**: Пользователь указывает уникальное имя и вставляет SVG-код.
2.  **Валидация**: На фронтенде проверяется уникальность имени в текущем списке `customIcons`.
3.  **Синхронизация**:
    - Фронтенд вызывает метод `addIcon()`.
    - Метод отправляет **полный обновленный список** иконок на сервер через `POST /icons`.
    - Бэкенд полностью перезаписывает файл `category-icons.json`.

```typescript
// Логика добавления на фронтенде
addIcon(icon: CategoryIconDto): void {
  const updated = [...this.state().customIcons, icon];
  this.api.saveIcons(updated).subscribe(() => {
    this.updateState({ customIcons: updated });
  });
}
```

## 4. Редактирование и удаление (Editing/Deletion)

### Редактирование
Поскольку иконки хранятся в простом массиве, редактирование реализовано как замена объекта в массиве с последующим сохранением всего списка через тот же эндпоинт `saveIcons()`.

### Удаление
1.  Метод `deleteIcon(name)` фильтрует массив `customIcons`, исключая иконку по имени.
2.  Обновленный массив отправляется на сервер.
3.  Бэкенд обновляет JSON-файл.

> [!WARNING]
> При удалении иконки, которая уже назначена какой-либо категории, категория перейдет на fallback-режим отображения (иконка по умолчанию или текст).

## 5. Использование в Категориях

Поле `IconUrl` в моделях `Category` и `Subcategory` содержит либо:
1.  **Имя кастомной иконки** (например, `av-games`) — в этом случае система ищет SVG в загруженном списке иконок.
2.  **Системное имя Ant Design** (например, `folder`).
3.  **Прямую ссылку** на внешний ассет.

Логика отображения инкапсулирована в компонентах, которые используют `iconsMap` из `CategorySimplifiedStateService` для мгновенного рендеринга SVG без дополнительных запросов к серверу.
