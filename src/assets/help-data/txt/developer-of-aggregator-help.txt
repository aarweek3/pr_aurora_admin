[ 📘 1. Общие положения ]
------------------------------------------
Модуль DeveloperOfAggregatorPage является частью системы справочников (SPRAVKA) раздела AGREGATOR. Он предназначен для централизованного управления базой данных разработчиков и вендоров программного обеспечения. Модуль обеспечивает полный цикл CRUD-операций, поддержку мультиязычного контента и SEO-параметров для каждого разработчика.

Путь в проекте: D:\_PROGECT\pr_aurora_admin\src\app\AGREGATOR\PAGES\SPRAVKA\DeveloperOfAggregatorPage

Назначение: Сущность DeveloperOfAggregator предназначена для хранения информации о создателях программного обеспечения. Она является ключевым справочник для группировки ПО по вендорам.

[ 🎨 Философия интерфейса Aurora v3.5 ]
------------------------------------------
Интерфейс спроектирован по паттерну "Information Density & Clarity". Мы используем мягкие тени и акцентные цвета для ключевых данных (ID, SystemCode), чтобы разработчик мог мгновенно ориентироваться в справочнике. Каждое действие имеет цветовую кодировку: синий — конгруэнтность, оранжевый — корзина, красный — сброс/очистка.

[ ⚙️ 2. Общий принцип работы ]
------------------------------------------
Архитектура: Модуль построен на архитектуре State Manager Pattern с использованием Angular Signals.
Управление данными: Все операции проходят через DeveloperOfAggregatorStateService.
Режимы отображения (Dual-Mode UI): Поддерживается два режима работы — «Модальный» (быстрое редактирование в окне) и «Страничный» (полноэкранная форма для глубокого редактирования).
Обслуживание БД: Предусмотрены инструменты для очистки таблицы и начального заполнения (seed) из JSON-файла.

Механизм English Fallback (Aurora v3.5 Standard)Система Aurora v3.5 минимизирует рутинную работу по заполнению переводов. Если пользователь не заполнил вкладку другого языка, данные будут автоматически подтянуты из эталонной английской версии.

Приоритеты Fallback:Значение в текущей вкладке.
Значение из вкладки en-US.

Охватываемые поля:Отображаемое название (Name)
Описание (Description)
SEO Title & Meta Description
TypeScript Logic:applyEnglishFallbacks(localizations: any[], technicalName: string): any[] {
  const enLocale = localizations.find(l => l.languageCode === 'en-US');
  return localizations.map(loc => ({
    ...loc,
    name: loc.name || enLocale?.name || technicalName,
    description: loc.description || enLocale?.description,
    metaTitle: loc.metaTitle || enLocale?.metaTitle,
    metaDescription: loc.metaDescription || enLocale?.metaDescription
  }));
}

[ ⚙️ Каскадный Hard Delete (Спецификация) ]
------------------------------------------
Логика безвозвратного удаления (Hard Delete)В отличие от Soft Delete, метод Hard Delete полностью очищает базу данных и файловую систему от упоминаний объекта. Это действие необратимо.

Алгоритм реализации (C# Service):Удаление локализаций: Сначала удаляются все записи из таблицы developer_of_aggregator_localizations.
Физическое удаление иконки: Если поле IconPath не пустое, необходимо удалить файл с диска, используя _webHostEnvironment.WebRootPath.
Удаление основной записи: В последнюю очередь удаляется сама сущность из developers_of_aggregator.
Пример реализации:public async Task HardDeleteAsync(int id)
{
    var entity = await _repository.GetByIdAsync(id);
    if (entity == null) return;

    // 1. Удаление файла
    if (!string.IsNullOrEmpty(entity.IconPath))
    {
        var fullPath = Path.Combine(_env.WebRootPath, entity.IconPath);
        if (File.Exists(fullPath)) File.Delete(fullPath);
    }

    // 2. Удаление из БД (Репозиторий сам удалит Localizations через Cascade)
    await _repository.HardDeleteAsync(entity);
    await _uow.SaveChangesAsync();
}

GetShard Method:private string GetShard(string filename)
{
    if (string.IsNullOrEmpty(filename)) return "unknown";
    char firstChar = char.ToLower(filename[0]);
    return (firstChar >= 'a' && firstChar <= 'z') ? firstChar.ToString() : "0";
}
