1.1 Концепция «Anchor Security» (DLL)

Anchor Security — это стратегический архитектурный паттерн, при котором вся критическая логика безопасности, аутентификации и авторизации вынесена в изолированную внешнюю библиотеку классов (Project.Aurora.Security.dll).

Почему это фундаментально важно:
  • Единая точка правды (SSOT): Все правила проверки паролей, политики JWT и механизмы работы с куками определены в одном месте. Любое изменение в DLL мгновенно распространяется на все API-сервисы проекта.
  • Dependency Inversion: Основные проекты приложения (например, Агрегатор) не зависят от тяжеловесных библиотек Microsoft Identity напрямую. Они получают готовую инфраструктуру через лаконичный метод расширения AddAuroraSecurity().
  • Изоляция инфраструктуры: Миграции базы данных Identity, модели пользователей и ролей живут внутри DLL, не смешиваясь с доменными моделями бизнеса.
  • Унификация сессий: Использование общей DLL позволяет легко реализовать Single Sign-On (SSO) между различными подсистемами Aurora в будущем.

1.2 Диаграмма потоков данных (Lifecycle)

Процесс авторизации в Aurora v3.5 разбит на 5 строго определенных технологических этапов:

ЭТАП 1: Инициация и Проверка (Login Flow)

Пользователь отправляет учетные данные. API вызывает сервис из DLL. Происходит обращение к физической базе AuroraIdentityDb. Проверяется не только пароль, но и статус блокировки (Lockout), а также подтверждение Email/Phone если это требуется политикой.

ЭТАП 2: Генерация "Двойного Токена" (Creation)

При успехе TokenService создает две разные сущности:
• JWT Access Token: Содержит Claims (ID, Name) и Роли. Подписывается секретным ключом. Живет недолго (15-30 мин).
• Opaque Refresh Token: Уникальный GUID, который связывается с записью в таблице UserSessions. Служит для продления жизни сессии без повторного ввода пароля.

ЭТАП 3: Безопасная "Посадка" (Cookie Storage)

Токены передаются клиенту исключительно через заголовки Set-Cookie. Используются флаги:
• HttpOnly: JavaScript (и вирусы) не имеют доступа к токену.
• Secure: Передача только по зашифрованному HTTPS.
• SameSite=Strict: Браузер не отправит куку при переходе со стороннего сайта (Защита от CSRF).

ЭТАП 4: Прозрачная Валидация (Verification)

При каждом запросе Middleware в DLL автоматически извлекает JWT из куки. Если подпись верна и срок не истек — HttpContext.User наполняется данными пользователя, и запрос проходит в контроллер.

ЭТАП 5: Бесшовное Продление (Silent Refresh)

Если JWT истек, фронтенд ловит 401 ошибку. Интерцептор (наш «невидимый герой») вызывает /api/Auth/refresh. Бэкенд проверяет Refresh-токен в базе данных и, если он жив, выдает новую пару кук. Все запросы пользователя повторяются автоматически.

1.3 Схема баз данных (DbNames vs IdentityDb)

В проекте реализована концепция Физической Изоляции Слой Безопасности.

AuroraIdentityDb

  • Таблицы: Пользователи, Роли, Клеймы, Сессии, Логи Аудита.
  • Технология: PostgreSQL (отдельная схема или инстанс).
  • Безопасность: Доступ строго ограничен. Никакая бизнес-логика не имеет права писать сюда напрямую.

DbNames (App Database)

  • Таблицы: Программы, Категории, Языки, Платформы и др.
  • Назначение: Хранение всего контента приложения.
  • Связь: Ссылается на UserId (GUID) как на внешнюю константу, не зная деталей профиля.
🛡️ Защита через "Blast Radius Reduction":

Если злоумышленник скомпрометирует основную базу данных (например, через SQL-инъекцию в поиске), он получит доступ к списку программ, но **физически не сможет добраться до хэшей паролей или активных сессий**, так как они находятся в другом контексте и другой базе данных с иными учетными данными доступа.

2.1 Модели данных (Entities & DTOs)

Основой безопасности являются расширенные модели ASP.NET Identity, адаптированные под нужды Aurora.

• ApplicationUser & ApplicationRole

Мы используем IdentityUser<Guid> для обеспечения глобальной уникальности идентификаторов. Роли поддерживают иерархическую структуру через Claims.

• Таблица UserSessions (Refresh Tokens)

Это "бортовой журнал" активных подключений. Каждая запись — это отдельное устройство или браузер пользователя. 

public class UserSession {
    public Guid Id { get; set; }           // Первичный ключ
    public string UserId { get; set; }     // Ссылка на пользователя
    public string RefreshToken { get; set; } // Opaque (непрозрачный) токен
    public DateTime CreatedAt { get; set; } // Дата входа
    public DateTime ExpiryDate { get; set; } // Срок жизни сессии
    public string IpAddress { get; set; }    // IP для защиты от угона
    public string UserAgent { get; set; }    // Браузер/ОС для аудита
    public bool IsRevoked { get; set; }      // Флаг принудительного отзыва
}

2.2 Сервисный слой (Core Logic)

• AuthService

Отвечает за высокоуровневые операции: Login, Register, Logout. Здесь происходит оркестрация: проверка пароля ➔ генерация токенов ➔ запись сессии в БД ➔ установка кук.

• TokenService

«Математический» центр DLL. Генерирует криптографически стойкие JWT токены.
Алгоритм: HmacSha256.
Payload: sub, jti, roles, exp.

• RoleService

Управляет правами доступа (Claims-Based Authorization). Позволяет динамически проверять, имеет ли пользователь право на конкретное действие в системе.

2.3 Конфигурация и Безопасность

Метод расширения AddAuroraSecurity() инкапсулирует всю сложность настройки. 

public static IServiceCollection AddAuroraSecurity(this IServiceCollection services, IConfiguration config) {
    // 1. Настройка Identity
    services.AddIdentity<ApplicationUser, ApplicationRole>(options => {
        options.Password.RequiredLength = 8;
        options.Lockout.MaxFailedAccessAttempts = 5; // Защита от Brute-force
    }).AddEntityFrameworkStores<AuroraIdentityDbContext>();

    // 2. Настройка JWT Bearer
    services.AddAuthentication()
            .AddJwtBearer(opt => {
                opt.TokenValidationParameters = new TokenValidationParameters {
                    ValidateIssuer = true,
                    ValidateAudience = true,
                    ValidateLifetime = true,
                    IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(config["Jwt:Key"]))
                };
            });

    // 3. Политика Cookie
    services.ConfigureApplicationCookie(options => {
        options.Cookie.HttpOnly = true;
        options.Cookie.SameSite = SameSiteMode.Strict;
        options.Cookie.SecurePolicy = CookieSecurePolicy.Always;
    });

    return services;
}

2.4 Логирование и Аудит

Безопасность невозможна без отслеживаемости. Мы фиксируем каждое событие в таблицу SecurityLogs.

  • Вход: Дата, успех/провал, причина отказа (неверный пароль/блокировка).
  • Смена пароля: Кто инициировал, IP адрес.
  • Подозрительная активность: Множественные запросы на Refresh токен с разных IP для одной сессии (детектирование кражи сессии).
Принцип минимизации данных: Мы логируем факты событий, но никогда не логируем чувствительные данные (пароли, старые токены и т.д.).

3.1 Структура модуля Auth

Модуль организован по стандарту Aurora для обеспечения высокой связности и низкой зависимости:

  • /services: Центральный AuthService и TokenStorageService.
  • /guards: AuthGuard (проверка входа) и RoleGuard (проверка прав).
  • /interceptors: AuthInterceptor (Silent Refresh).
  • /models: Интерфейсы User, AuthResponse, UserRole.

3.2 State Management (Signals)

Мы используем Angular Signals для мгновенного и эффективного обновления UI при изменении состояния авторизации. 

@Injectable({ providedIn: 'root' })
export class AuthService {
  // Базовый сигнал текущего пользователя
  private _currentUser = signal<User | null>(null);
  
  // Экспортируем как readonly
  public currentUser = this._currentUser.asReadonly();

  // Computed сигналы для логики
  public isAuthenticated = computed(() => !!this._currentUser());
  public isAdmin = computed(() => this._currentUser()?.roles?.includes('Admin') ?? false);
  public userRoles = computed(() => this._currentUser()?.roles ?? []);

  // Метод обновления состояния
  public setUser(user: User | null) {
    this._currentUser.set(user);
  }
}

3.3 Silent Refresh: Interceptor (The Core)

Самый сложный узел фронтенда. Интерцептор перехватывает 401 ошибку, ставит все текущие запросы на паузу, обновляет токены и повторяет запросы без участия пользователя. 

private isRefreshing = false;
private refreshTokenSubject: BehaviorSubject<any> = new BehaviorSubject<any>(null);

intercept(req: HttpRequest<any>, next: HttpHandler): Observable<HttpEvent<any>> {
  return next.handle(req).pipe(
    catchError(error => {
      if (error instanceof HttpErrorResponse && error.status === 401) {
        return this.handle401Error(req, next);
      }
      return throwError(() => error);
    })
  );
}

private handle401Error(request: HttpRequest<any>, next: HttpHandler) {
  if (!this.isRefreshing) {
    this.isRefreshing = true;
    this.refreshTokenSubject.next(null);

    return this.authService.refreshToken().pipe(
      switchMap(() => {
        this.isRefreshing = false;
        this.refreshTokenSubject.next(true);
        return next.handle(request); // Повторяем упавший запрос
      }),
      catchError((err) => {
        this.isRefreshing = false;
        this.authService.logout(); // Если рефреш не удался - выкидываем
        return throwError(() => err);
      })
    );
  } else {
    // Если рефреш уже идет - ждем его завершения через BehaviorSubject
    return this.refreshTokenSubject.pipe(
      filter(token => token !== null),
      take(1),
      switchMap(() => next.handle(request))
    );
  }
}

3.4 Интеграция с UI

Использование сигналов в компонентах позволяет писать чистый и декларативный код.

// В шаблоне компонента
<div *if="auth.isAuthenticated()">

  <button *if="auth.isAdmin()" nz-button>Панель управления</button>
</div>

4.1 Эшелонированная защита (XSS & CSRF)

Мы не полагаемся на один механизм, а используем комбинацию защитных мер на уровне протокола HTTP:

• Защита от XSS (Cross-Site Scripting)

Все токены (Access и Refresh) передаются исключительно через HttpOnly Cookies. Это означает, что даже если злоумышленник внедрит вредоносный скрипт на страницу, он физически не сможет прочитать токены через document.cookie.

• Защита от CSRF (Cross-Site Request Forgery)

Использование флага SameSite=Strict инструктирует браузер отправлять куки только при запросах, инициированных с нашего собственного домена. Любые запросы со сторонних ресурсов будут приходить без авторизационных данных.

4.2 Ротация токенов (Token Rotation)

Для минимизации ущерба при компрометации Refresh-токена мы используем стратегию ротации:

  • При каждом вызове метода /refresh старый Refresh-токен аннулируется (удаляется из таблицы UserSessions).
  • Клиенту выдается абсолютно новая пара токенов.
  • Если кто-то попытается использовать старый (уже использованный) Refresh-токен, система расценивает это как атаку и может мгновенно аннулировать все активные сессии данного пользователя.
Важно: Это предотвращает бесконечный доступ злоумышленника к аккаунту, даже если он смог один раз перехватить токен.

4.3 Валидация и защита от перебора

Настройки сложности пароля и механизмов блокировки задаются в конфигурации Identity внутри DLL:

options.Password.RequiredLength = 8;
options.Password.RequireNonAlphanumeric = true; // Спецсимволы
options.Password.RequireUppercase = true;       // Заглавные буквы
options.Password.RequireLowercase = true;       // Строчные буквы

// Защита от Brute-force
options.Lockout.DefaultLockoutTimeSpan = TimeSpan.FromMinutes(15);
options.Lockout.MaxFailedAccessAttempts = 5; // Блокировка после 5 попыток
options.Lockout.AllowedForNewUsers = true;
Аудит блокировок: Каждая блокировка аккаунта сопровождается записью в лог безопасности для последующего анализа администратором.
Защита контроллера:
[Authorize(Roles = "Admin, Editor")]
public class MyController : ControllerBase { ... }
Проверка роли во Фронтенде:
@if (authService.isAdmin()) {
  <app-admin-panel />
}
Добавление роли: 1. Вставить запись в AspNetRoles. 2. Обновить Enum UserRole в Angular.

6.1 Разбор критических ошибок

Q: Почему я получаю 401 Unauthorized, если я только что залогинился?

Возможные причины:

  • Time Skew: Время на сервере и клиенте разошлось более чем на 5 минут (JWT валидация по умолчанию учитывает 5 мин дрейфа).
  • Cookie Domain: Приложение и API запущены на разных доменах без правильной настройки CORS и WithCredentials.
  • Token Expiry: Refresh-токен был отозван в базе данных (например, при входе с другого устройства).
Q: Разница между 401 и 403?
• 401 (Unauthorized): Система не знает, кто вы (невалидный токен).
• 403 (Forbidden): Система знает, кто вы, но у вас нет нужной Роли для этого действия.

6.2 Управление сессиями

• Как сбросить все сессии пользователя?

Для мгновенного разлогина пользователя на всех устройствах необходимо очистить таблицу UserSessions: 

DELETE FROM "UserSessions" WHERE "UserId" = 'uuid-пользователя';

* При следующем запросе фронтенд получит 401, не сможет сделать Refresh и перенаправит на страницу логина.

6.3 Отказоустойчивость

• Что делать, если база IdentityDb недоступна?

Система авторизации Aurora спроектирована по принципу Fail-Fast. Если база безопасности недоступна, API будет возвращать статус 500 Internal Server Error. Работа приложения в анонимном режиме при наличии защищенных роутов запрещена по соображениям безопасности.

6.4 Чек-лист: "Если ничего не работает"

  1. Проверьте вкладку Application -> Cookies в инструментах разработчика. Есть ли там куки accessToken и refreshToken?
  2. Проверьте Console на наличие ошибок CORS.
  3. Убедитесь, что в БД в таблице UserSessions появилась запись после логина.
  4. Проверьте системные логи API на наличие исключений при инициализации AddAuroraSecurity().
  5. Очистите куки вручную и попробуйте зайти снова.
