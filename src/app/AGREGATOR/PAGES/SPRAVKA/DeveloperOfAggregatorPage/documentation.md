# Документация: Модуль "Разработчики Агрегатора" (DeveloperOfAggregatorPage)

## 1. Общие положения
Модуль **DeveloperOfAggregatorPage** является частью системы справочников (SPRAVKA) раздела AGREGATOR. Он предназначен для централизованного управления базой данных разработчиков и вендоров программного обеспечения. Модуль обеспечивает полный цикл CRUD-операций, поддержку мультиязычного контента и SEO-параметров для каждого разработчика.

## 2. Общий принцип работы
Модуль построен на архитектуре **State Manager Pattern** с использованием Angular Signals. 
*   **Управление данными**: Все операции проходят через `DeveloperOfAggregatorStateService`, который синхронизирует состояние UI с API.
*   **Режимы отображения**: Поддерживается два режима работы — «Модальный» (быстрое редактирование в окне) и «Страничный» (полноэкранная форма).
*   **Обслуживание БД**: Предусмотрены инструменты для очистки таблицы и начального заполнения (seed) из JSON-файла.

## 3. Локализация и SEO
Для каждого разработчика поддерживается мультиязычность. Каждая локализация включает:
*   Публичное название (Name).
*   Подробное описание (Description).
*   SEO-метаданные (Meta Title, Meta Description).
Система автоматически применяет английские fallback-значения для пустых полей других языков.

## 5. Блоки для Универсальной Справки (JSON)

### Блок: Общая информация
**Разработчики Агрегатора** — это справочник вендоров, чье ПО представлено в системе. Каждая запись связывается с программными продуктами через `DeveloperId`.

### Блок: Работа с иконками
Система использует **ImageServiceUniversal**. Иконки разработчиков хранятся в папке `developers/`. При загрузке иконки через форму она автоматически попадает в нужную директорию на сервере.

### Блок: Режимы ViewMode
Вы можете переключаться между режимами «Модалка» и «Страница» в шапке менеджера. Режим «Модалка» удобен для быстрой правки, режим «Страница» — для работы с большим объемом текстового описания и SEO.

## 6. Технические подробности: Блок обслуживания (Maintenance)

Для управления состоянием базы данных используется выносной компонент управления JSON-данными.

*   **Компонент**: `ButtonControlJsonBlockComponent`
*   **Путь**: `@shared/controls/button-control-json-block/`
*   **Связь с Manager**:
    *   **Активация**: Кнопка настроек в хедере переключает флаг `showMaintenance`.
    *   **Анимация**: Иконка шестеренки имеет привязку `[nzSpin]="showMaintenance"`.
    *   **Интеграция**: Компонент подключается через `*ngIf`, принимает `[loading]` и `[total]` от стейт-сервиса и транслирует события `onClear`, `onRead`, `onSeed` обратно в менеджер для выполнения соответствующих действий в `DeveloperOfAggregatorStateService`.

## 7. Поиск и Список (Search & List)

Механика фильтрации данных реализована через связку двух компонентов и стейт-сервиса.

*   **AvSearchComponent** (Shared):
    *   Предоставляет UI ввода с иконкой очистки и индикатором загрузки.
    *   Связывается через `[(value)]` и событие `(onSearch)`.
*   **DeveloperOfAggregatorListComponent** (Local):
    *   Слушает событие `onSearch` и транслирует его в `state.setSearch(term)`.
    *   Отображает таблицу `nz-table` с данными из `state.items()`.
*   **Взаимодействие со State**:
    *   При вызовете `setSearch()` сервис сбрасывает `pageNumber` в 1 и вызывает `loadItems()`.
    *   Это обеспечивает "бесшовную" фильтрацию данных в реальном времени.

## 8. Форма и Табуляция (Lazy Form Architecture)

Для работы с большим количеством языков и сложными SEO-данными используется паттерн ленивой инициализации.

*   **Lazy Tabs**:
    *   Реализовано через `nz-tabset` and `ng-template nz-tab`.
    *   DOM-дерево вкладки создается только при активации пользователем. Это снижает нагрузку на браузер при инициализации формы с 10+ языками.
*   **Data Mapping (SEO)**:
    *   **Frontend-структура**: Вложенный объект `seoData` для работы с `SeoFormComponent`.
    *   **Backend-структура**: Плоские поля `metaTitle`, `metaDescription` в таблице локализаций.
    *   **Трансформация**: Выполняется в методах `loadData` (маппинг вглубь) и `submit` (развертывание наружу перед сохранением).

## 9. Корзина (Recycle Bin / Soft Delete)

Модуль поддерживает стандартный для Aurora v3.5 механизм мягкого удаления.

*   **Переключатель**: Находится в хедере списка. Связан с `state.showDeleted()`.
*   **API**: При активации в GET-запрос добавляется параметр `showDeleted=true`.
*   **UI-состояние**:
    *   **Alert**: Вверху списка отображается `nz-alert` о режиме корзины.
    *   **Contextual Actions**:
        *   `onRestore(id)`: Вызывает POST `restore`. Запись возвращается в основной список.
        *   `onHardDelete(id)`: Вызывает DELETE с параметром `isHard=true`. Требует прохождения `modal.challenge`.
