## 2. Команды генерации (CLI)

### Шаг 0: Создание папки и воркспейса

Выполняется из корня `D:/AURORA/`. Используем явный флаг `--standalone` для надежности.

```bash
mkdir D:/AURORA
cd D:/AURORA

npx create-nx-workspace@latest aggregator-web \
  --preset=angular \
  --appName=admin \
  --standalone \
  --style=css \
  --bundler=vite \
  --nxCloud=skip
```

**Расшифровка флагов:**

- `aggregator-web`: Имя папки воркспейса.
- `--preset=angular`: Создание проекта на базе Angular.
- `--appName=admin`: Название основного приложения управления.
- `--standalone`: Использование Standalone-компонентов (без `AppModule`).
- `--style=css`: Использование чистого CSS (стандарт Aurora).
- `--bundler=vite`: Использование сверхбыстрого сборщика Vite.

**Что получится в результате выполнения этой команды:**

1. **На диске**: Создастся новая папка `aggregator-web`. Внутри неё вы увидете папки `apps` (уже с подпапкой `admin`) и `libs` (пока пустую), а также конфигурационные файлы (`nx.json`, `package.json`).
2. **В терминале**: Nx установит все необходимые зависимости (npm install). Это может занять пару минут.
3. **В коде**: У вас будет готовое к запуску приложение `admin`. В файле `tsconfig.base.json` уже будет настроен базовый путь для ваших будущих библиотек.
4. **Готовность**: Вы сможете сразу зайти в папку (`cd aggregator-web`) и запустить проект командой `npx nx serve admin`. Вы увидете пустую стартовую страницу Angular, работающую на Vite.

### Шаг 1: Создание приложения Site с поддержкой SSR

Для агрегатора критично SEO, поэтому включаем SSR сразу.

```bash
cd aggregator-web
nx g @nx/angular:app site --directory=apps/site --style=css --bundler=vite --ssr
```

**Что получится:**

- В папке `apps/` появится вторая директория `site`.
- Это полностью независимое приложение со своим `index.html`.
- Теперь вы сможете запускать параллельно и админку (`nx serve admin`), и сайт (`nx serve site`).

### Шаг 1.1: Ручная настройка тегов для приложений (ВАЖНО)

Чтобы заработали архитектурные границы (Boundary Rules), нужно открыть `project.json` каждого приложения и добавить теги в **корень** файла.

**Пример для apps/admin/project.json:**

```json

  "tags": ["type:app", "domain:admin"],
```

**Аналогично для apps/site/project.json:** `"tags": ["type:app", "domain:site"]`.

### Шаг 2: Генерация инфраструктуры Shared


**Что получится:**

- В папке `libs/shared/` появятся 4 новые директории.
- В каждой из них будет файл `src/index.ts` — это «лицо» библиотеки (Public API).
- **Магия Nx**: В файле `tsconfig.base.json` появятся пути типа `@aggregator/shared/models`. Вы сможете импортировать код из этих папок в любое приложение (`admin` или `site`) через этот красивый путь.

### Шаг 3: Генерация доменов (Auth & Aggregator)

```bash
# Auth
nx g @nx/angular:lib data-access --directory=libs/auth/data-access --tags="type:data-access,domain:auth"
nx g @nx/angular:lib feature-admin --directory=libs/auth/feature-admin --tags="type:feature,domain:auth"
nx g @nx/angular:lib feature-site --directory=libs/auth/feature-site --tags="type:feature,domain:auth"

# Aggregator
nx g @nx/angular:lib data-access --directory=libs/aggregator/data-access --tags="type:data-access,domain:aggregator"
nx g @nx/angular:lib feature-details --directory=libs/aggregator/feature-details --tags="type:feature,domain:aggregator"
nx g @nx/angular:lib feature-list --directory=libs/aggregator/feature-list --tags="type:feature,domain:aggregator"
```

**Что получится:**

- Создастся структура для авторизации и бизнеса в `libs/`.
- Модуль `data-access` станет местом, где живет логика API и сервисов.
- Вы сможете подключать общий код к обоим приложениям одновременно.

## 4. Технические риски и решения

### Риск 1: TinyMCE Assets (Критично для Admin и Site)

**Проблема**: Редактор требует статические ассеты (скины, иконки) из `node_modules`.
**Решение**: В `apps/admin/project.json` и `apps/site/project.json` добавить явный относительный путь:

```json
"assets": [
  { "glob": "**/*", "input": "../../node_modules/tinymce", "output": "/tinymce" }
]
```

### Риск 2: SSR + Browser-only API (Критично для Site)

**Проблема**: Любой код, обращающийся к `window`, `document`, `localStorage` или `ResizeObserver` (включая TinyMCE), приведет к падению приложения на сервере.
**Решение**: Использовать `isPlatformBrowser` из `@angular/common` для изоляции кода:

```typescript
if (isPlatformBrowser(this.platformId)) {
  // Код, использующий window/document
}
```

## 5. Соглашения по именованию и структуре

### Public API (index.ts)

## 7. Порядок миграции (Pull Strategy)

1. **Shared Models**: Перенос всех интерфейсов и DTO.
2. **Shared Util**: Пайпы и общие хелперы.
3. **Shared UI-Editors**: TinyMCE + настройка ассетов в `project.json`.
4. **Auth Data-Access**: Сервисы авторизации и интерцепторы.
5. **Auth Features**: Страницы логина для админки и сайта.
6. **Shared UI-Media**: Блок работы с иконками и скриншотами.
7. **Aggregator Data-Access**: Основные сервисы данных.
8. **Aggregator Features**: Списки и детальные страницы.
