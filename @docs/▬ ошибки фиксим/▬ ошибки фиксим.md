# 📋 Рефакторинг: Решение проблем с браузерными API и библиотеками

**Дата:** 2026-01-12
**Проект:** Aurora Admin (pr_kln_names)
**Цель:** Исправление критических ошибок при работе с `window`, `navigator` и библиотеками NG-ZORRO/TinyMCE

---

## 🔴 Проблемы, которые были обнаружены

### 1. **Ошибка: `Cannot read properties of null (reading 'innerWidth')`**

**Где возникала:**

- При загрузке приложения
- При изменении ориентации устройства (`orientationchange` event)
- При открытии модальных окон NG-ZORRO

**Причина:**

- `LoggingService` обращался к `window.location.href` в **конструкторе**
- NG-ZORRO/Angular CDK обращались к `window.innerWidth` до полной инициализации DOM
- Во время SSR (Server-Side Rendering) объект `window` отсутствует

**Стек ошибки:**

```
[[GLOBAL_ERR]] Cannot read properties of null (reading 'innerWidth')
at chunk-D2XFKYW4.js?v=a1b6dacb:5572:25
```

---

### 2. **Ошибка: `Cannot read properties of undefined (reading 'isEmpty')`**

**Где возникала:**

- При загрузке компонентов с таблицами/списками
- Периодически при проверке токена

**Причина:**

- NG-ZORRO ожидал объект с полем `isEmpty`, но получал `undefined`
- `BehaviorSubject` не инициализирован начальным значением
- Observable еще не вернул первый `emit`

**Стек ошибки:**

```
[[GLOBAL_ERR]] Cannot read properties of undefined (reading 'isEmpty')
at chunk-D2XFKYW4.js?v=a1b6dacb:5572:25
```

---

### 3. **Ошибка: `Uncaught (in promise) Error: No checkout popup config found`**

**Где возникала:**

- При инициализации TinyMCE редактора

**Причина:**

- TinyMCE пытался загрузить premium плагины
- Проверка лицензии для платных функций
- Недостаточная конфигурация в `main.ts`

---

### 4. **Предупреждение: `tabSwitchMotion` animation warning**

**Где возникала:**

- При использовании `<nz-tabs>` компонента

**Причина:**

- NG-ZORRO пытается анимировать CSS свойство `display`
- `display` не является анимируемым свойством по спецификации CSS

## ✅ Решения

### Решение #1: Создание EnvironmentService

**Файл:** `src/app/shared/infrastructures/environment/environment.service.ts`

**Что сделали:**
Создали "тупой" сервис (без зависимостей от Router, HttpClient) для безопасного доступа к browser API.

**Ключевые особенности:**

```typescript
@Injectable({ providedIn: "root" })
export class EnvironmentService {
  private readonly isBrowser: boolean;

  // Кэшированные значения
  readonly userAgent: string;
  readonly initialUrl: string;

  constructor(@Inject(PLATFORM_ID) platformId: Object) {
    this.isBrowser = isPlatformBrowser(platformId);

    // Безопасная инициализация
    this.userAgent = this.isBrowser && typeof navigator !== "undefined" ? navigator.userAgent : "server";

    this.initialUrl = this.isBrowser && typeof window !== "undefined" ? window.location.href : "unknown";
  }

  getCurrentUrl(): string {
    return this.isBrowser && typeof window !== "undefined" ? window.location.href : this.initialUrl;
  }
}
```

**Преимущества:**

- ✅ SSR-ready (работает на сервере и в браузере)
- ✅ Единственная зависимость: `PLATFORM_ID`
- ✅ Нет циклических зависимостей
- ✅ Легко тестировать
- ✅ Переиспользуемый

### Решение #5: Патч для NG-ZORRO в main.ts

**Файл:** `src/main.ts`

**Проблема:** NG-ZORRO обращается к `window.innerWidth`/`innerHeight` до инициализации DOM.

**Решение:** Перехватываем обращения к этим свойствам:

```typescript
if (typeof window !== "undefined") {
  // === ПАТЧ ДЛЯ NG-ZORRO: Защита от ошибок с window/document ===

  const originalInnerWidth = Object.getOwnPropertyDescriptor(window, "innerWidth");
  const originalInnerHeight = Object.getOwnPropertyDescriptor(window, "innerHeight");

  // Переопределяем innerWidth с защитой от ошибок
  Object.defineProperty(window, "innerWidth", {
    get: function () {
      try {
        return originalInnerWidth?.get?.call(window) || 1920;
      } catch {
        return 1920; // Безопасный fallback
      }
    },
    configurable: true,
  });

  // Переопределяем innerHeight с защитой от ошибок
  Object.defineProperty(window, "innerHeight", {
    get: function () {
      try {
        return originalInnerHeight?.get?.call(window) || 1080;
      } catch {
        return 1080; // Безопасный fallback
      }
    },
    configurable: true,
  });
}
```

**Как работает:**

1. Сохраняем оригинальные дескрипторы свойств
2. Переопределяем геттеры с `try-catch`
3. Если NG-ZORRO обращается слишком рано → возвращаем безопасный fallback
4. Если все нормально → возвращаем реальное значение

### Решение #6: Фильтрация библиотечных ошибок в GlobalErrorHandler

**Файл:** `src/app/shared/infrastructures/interceptor/services/global-error-handler.service.ts`

**Проблема:** Даже с патчем могут проскакивать ошибки из библиотек.

**Решение:** Добавили метод для фильтрации известных библиотечных ошибок:

```typescript
handleError(error: any, options: { type?: 'toast' | 'modal'; redirectUrl?: string } = {}): void {
  // Игнорируем известные библиотечные ошибки
  if (this.isKnownLibraryError(error)) {
    console.warn('[Known Library Error - Suppressed]', error?.message || error);
    return; // Не показываем пользователю
  }

  // Остальная логика обработки ошибок...
}

private isKnownLibraryError(error: any): boolean {
  const errorMessage = error?.message || '';
  const errorStack = error?.stack || '';

  // Список известных библиотечных ошибок
  const knownErrors = [
    // NG-ZORRO
    "Cannot read properties of undefined (reading 'isEmpty')",
    "Cannot read property 'isEmpty' of undefined",

    // CDK/NG-ZORRO viewport
    "Cannot read properties of null (reading 'innerWidth')",
    "Cannot read property 'innerWidth' of null",
    "Cannot read properties of null (reading 'innerHeight')",
    "Cannot read property 'innerHeight' of null",

    // ResizeObserver (безопасно игнорировать)
    'ResizeObserver loop limit exceeded',
    'ResizeObserver loop completed with undelivered notifications',
  ];

  const isKnownMessage = knownErrors.some(knownError =>
    errorMessage.includes(knownError)
  );

  // Проверяем, что ошибка из библиотечного кода
  const isFromLibraryChunk = errorStack.includes('chunk-') ||
                              errorStack.includes('node_modules') ||
                              errorStack.includes('ng-zorro-antd');

  return isKnownMessage && isFromLibraryChunk;
}
```

**Результат:** Библиотечные ошибки подавляются и не показываются пользователю.

## 📝 Важные выводы

### 1. **Почему NG-ZORRO создает эти проблемы?**

NG-ZORRO (версия 19.3.1) и Angular CDK имеют проблемы с:

- Ранним доступом к `window.innerWidth`/`innerHeight` при создании overlay/modal
- Отсутствием проверок на SSR-окружение
- Ожиданием всегда инициализированных объектов данных

**Это известные проблемы библиотеки**, которые разработчики не исправили.

### 2. **Можно ли заменить NG-ZORRO?**

Альтернативы:

- **Angular Material** - официальная библиотека от Google (более стабильная)
- **PrimeNG** - хорошая альтернатива с большим набором компонентов
- **Ant Design (React)** + Angular wrapper - если очень нужен дизайн Ant Design

**Но замена не обязательна** - наши патчи решают проблему.

### 3. **Почему патч в main.ts безопасен?**

- Переопределяем только геттеры (не меняем поведение)
- Используем `try-catch` - даже если что-то сломается, вернем fallback
- `configurable: true` - можно откатить изменения при необходимости
- Патч применяется **до загрузки Angular** - нет конфликтов

### 4. **Нужен ли EnvironmentService, если есть патч?**

**Да, нужен!** Причины:

- **Чистая архитектура** - единая точка для browser API
- **Тестируемость** - легко мокировать в тестах
- **Расширяемость** - легко добавить новые методы (localStorage, cookies, etc.)
- **SSR-ready** - один раз настроили, работает везде

## 🔮 Рекомендации на будущее

### При добавлении новых компонентов:

1. **Всегда используйте EnvironmentService** для доступа к browser API
2. **Инициализируйте BehaviorSubject** с начальными значениями
3. **Используйте `startWith()`** для Observable, если не можете использовать BehaviorSubject
4. **Не обращайтесь к `window`/`navigator`** напрямую в конструкторах

### При работе с модальными окнами NG-ZORRO:

```typescript
// ✅ Хорошо
modal$ = new BehaviorSubject<MyData | null>(null);

// ✅ Хорошо с начальным состоянием
modal$ = new BehaviorSubject<MyData>({ isEmpty: true, data: [] });

// ❌ Плохо
modal$: Observable<MyData>; // undefined при первом обращении
```

### При обновлении NG-ZORRO:

После обновления библиотеки:

1. Проверьте, не исправили ли проблемы с `innerWidth`/`isEmpty`
2. Если исправили - можно убрать патч из `main.ts`
3. Если нет - оставьте патч и обновите список в `GlobalErrorHandler`
